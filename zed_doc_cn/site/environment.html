环境变量

注意：以下内容仅适用于 Zed 0.152.0 及更高版本。

Zed 中的多个功能受环境变量影响：

  • 任务
  • 内置终端
  • 语言服务器查找
  • 语言服务器

为了更好地使用这些功能，了解 Zed 从哪里获取环境变量以及如何使用它们是很有帮助的。

Zed 从哪里获取环境变量？

Zed 的启动方式 —— 是通过点击 macOS 的 Dock 图标或 Linux 窗口管理器中的图标，还是通过 Zed 自带的 CLI zed 命令启动 —— 会影响 Zed 可以使用的环境变量。

从 CLI 启动

如果通过 CLI（zed）打开 Zed，它将从周围的 shell 会话中继承环境变量。

这意味着如果您执行

$ export MY_ENV_VAR=hello
$ zed .

环境变量 MY_ENV_VAR 现在在 Zed 内部可用。例如，在内置终端中。

从 Zed 0.152.0 开始，CLI zed 将始终将其环境传递给 Zed，无论之前是否有 Zed 实例正在运行。在 Zed 0.152.0 之前情况并非如此，只有第一个 Zed 实例才会继承环境变量。

通过窗口管理器、Dock 或启动器启动

当通过 macOS 的 Dock、Linux 上的 GNOME 或 KDE 图标，或者像 Alfred 或 Raycast 这样的应用启动器启动 Zed 时，它没有周围的 shell 环境来继承环境变量。

为了仍然拥有有用的环境，Zed 会在用户的主目录中生成一个登录 shell 并获取其环境。然后这个环境被设置在 Zed 进程上。这意味着所有 Zed 窗口和项目都将继承该主目录环境。

由于这可能会为需要项目的不同环境变量的用户带来问题（因为他们在该项目中使用 direnv、asdf、mise 等），当打开项目时，Zed 会生成另一个登录 shell。这次是在项目的目录中。来自该登录 shell 的环境不会设置在进程上（因为这意味着打开一个新项目会改变所有 Zed 窗口的环境）。相反，环境被存储并在运行任务、打开终端或生成语言服务器时传递。

环境变量在哪里以及如何使用？

有两组环境变量：

  1. Zed 进程的环境变量
  2. 每个项目存储的环境变量

来自 (1) 的变量始终被使用，因为它们存储在进程本身上，每个生成的进程（任务、终端、语言服务器等）默认情况下都会继承它们。

来自 (2) 的变量会根据功能显式使用。

任务

任务使用组合环境生成。按优先级顺序（从低到高，后者覆盖前者）：

  • Zed 进程环境
  • 如果项目是从 CLI 打开的：CLI 环境
  • 如果项目不是从 CLI 打开的：通过在项目根目录中运行登录 shell 获得的项目环境变量
  • 可选，在设置中显式配置的环境

内置终端

内置终端与任务一样，使用组合环境生成。按优先级顺序（从低到高）：

  • Zed 进程环境
  • 如果项目是从 CLI 打开的：CLI 环境
  • 如果项目不是从 CLI 打开的：通过在项目根目录中运行登录 shell 获得的项目环境变量
  • 可选，在设置中显式配置的环境

语言服务器查找

对于某些语言，语言服务器适配器会在用户的 $PATH 中查找二进制文件。例如：

  • Go
  • Zig
  • Rust（如果配置为如此）
  • C
  • TypeScript

对于这个查找，Zed 使用以下环境：

  • 如果项目是从 CLI 打开的：CLI 环境
  • 如果项目不是从 CLI 打开的：通过在项目根目录中运行登录 shell 获得的项目环境变量

语言服务器

在查找到语言服务器后，Zed 会启动它们。

这些语言服务器进程始终继承 Zed 的进程环境。但是，根据语言服务器查找的情况，可能会设置额外的环境变量或覆盖进程环境。

  • 如果在项目环境的 $PATH 中找到了语言服务器，那么项目环境会传递给语言服务器进程。项目环境的来源取决于项目是如何打开的，是通过 CLI 还是其他方式。请参阅上一点关于语言服务器查找的内容。
  • 如果在项目环境中找不到语言服务器，Zed 会尝试全局安装并全局启动它。在这种情况下，进程将继承 Zed 的进程环境，并且 —— 如果项目是通过 CLI 打开的 —— 还会从 CLI 继承。
