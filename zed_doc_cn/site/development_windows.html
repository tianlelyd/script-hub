在 Windows 上构建 Zed

以下命令可以在任意 shell 中执行。

代码仓库

克隆 Zed 代码仓库。

依赖项

  • 安装 rustup

  • 安装 Visual Studio，需要包含可选组件 MSVC v*** - VS YYYY C++ x64/x86 build tools 和 MSVC v*** - VS YYYY C++ x64/x86 Spectre-mitigated libs (latest)（v*** 是您的 VS 版本，YYYY 是 VS 发布年份。请注意架构，如需要请更改为您的架构。）

  • 或者，如果您希望使用更精简的安装程序来只安装 MSVC 编译器工具，您可以安装 构建工具（加上上述库）和 "使用 C++ 的桌面开发" 工作负载。 但是请注意，这个安装不会被 rustup 自动检测到。您必须先启动该安装在开始菜单或 Windows Terminal 中放置的 "开发者" shell（cmd/powershell）来初始化您的环境变量，然后再进行编译。

  • 根据您的系统安装 Windows 11 或 10 SDK，但请确保您的机器上至少安装了 Windows 10 SDK version 2104 (10.0.20348.0)。您可以从 Windows SDK 存档 中下载它

  • 安装 CMake（由 依赖项 所需）。或者您可以通过 Visual Studio Installer 安装它，然后手动将 bin 目录添加到您的 PATH 中，例如：C:\Program Files\Microsoft Visual Studio\2022\Community\Common7\IDE\CommonExtensions\Microsoft\CMake\CMake\bin。

如果您无法编译 Zed，请确保在 Visual Studio 安装情况下您至少安装了以下组件：

后端依赖项（可选）

如果您希望使用本地协作服务器开发 Zed 协作功能，请参阅：本地协作 文档。

注意事项

您应该修改 data 目录中的 pg_hba.conf 文件，将 host 方法从 scram-sha-256 改为 trust。否则，连接将失败并显示错误 password authentication failed。pg_hba.conf 文件通常位于 C:\Program Files\PostgreSQL\17\data\pg_hba.conf。修改后，文件应该如下所示：

# IPv4 local connections:
host    all             all             127.0.0.1/32            trust
# IPv6 local connections:
host    all             all             ::1/128                 trust

此外，如果您使用非拉丁字符的 Windows 版本，您必须在 data 目录中的 postgresql.conf 文件中将 lc_messages 参数修改为 English_United States.1252（或您拥有的任何 UTF8 兼容编码）。否则，数据库将会崩溃。postgresql.conf 文件应该如下所示：

# lc_messages = 'Chinese (Simplified)_China.936' # locale for system error message strings
lc_messages = 'English_United States.1252'

之后，您应该重启 postgresql 服务。按 win 键 + R 打开 运行 窗口。输入 services.msc 并点击 确定 按钮打开服务管理器。然后，找到 postgresql-x64-XX 服务，右键点击它，并选择 重启。

从源码构建

一旦您安装了依赖项，您就可以使用 Cargo 构建 Zed。

进行调试构建：

cargo run

进行发布构建：

cargo run --release

运行测试：

cargo test --workspace

从 msys2 安装

Zed 不支持为 Mingw-w64 构建的非官方 MSYS2 Zed 包。如果您在使用 mingw-w64-zed 时遇到任何问题，请向 msys2/MINGW-packages/issues 报告。

请先参阅 MSYS2 文档。

故障排除

设置 RUSTFLAGS 环境变量会破坏构建

如果您设置了 RUSTFLAGS 环境变量，它将覆盖 .cargo/config.toml 中的 rustflags 设置，而这些设置是正确构建 Zed 所必需的。

由于这些设置可能会不时变化，您收到的构建错误可能从链接器错误到其他更奇怪的错误都有。

如果您希望添加额外的 rust 标志，您可以在 .cargo/config.toml 中执行以下一种操作：

在 build 部分中添加您的标志

[build]
rustflags = ["-C", "symbol-mangling-version=v0", "--cfg", "tokio_unstable"]

在 windows 目标部分中添加您的标志

[target.'cfg(target_os = "windows")']
rustflags = [
    "--cfg",
    "windows_slim_errors",
    "-C",
    "target-feature=+crt-static",
]

或者，您可以在与 Zed 代码仓库相同的文件夹中创建一个新的 .cargo/config.toml（见下文）。如果您进行 CI 构建，这特别有用，因为您不必编辑原始的 .cargo/config.toml。

upper_dir
├── .cargo          // <-- Make this folder
│   └── config.toml // <-- Make this file
└── zed
    ├── .cargo
    │   └── config.toml
    └── crates
        ├── assistant
        └── ...

在新的（上述）.cargo/config.toml 中，如果我们想在 rustflags 中添加 --cfg gles，它将如下所示

[target.'cfg(all())']
rustflags = ["--cfg", "gles"]

Cargo 错误声称依赖项使用了不稳定功能

请尝试 cargo clean 和 cargo build。

STATUS_ACCESS_VIOLATION

如果您使用的是 "rust-lld.exe" 链接器，就可能出现此错误。请考虑尝试不同的链接器。

如果您使用的是全局配置，请考虑将 Zed 代码仓库移动到嵌套目录，并在父目录中添加一个带有自定义链接器配置的 .cargo/config.toml。

有关更多信息，请参见此问题 #12041

无效的 RC 路径选择

有时，根据适用于您笔记本电脑的安全规则，您在编译 Zed 时可能会遇到以下错误：

error: failed to run custom build command for `zed(C:\Users\USER\src\zed\crates\zed)`

Caused by:
  process didn't exit successfully: `C:\Users\USER\src\zed\target\debug\build\zed-b24f1e9300107efc\build-script-build` (exit code: 1)
  --- stdout
  cargo:rerun-if-changed=../../.git/logs/HEAD
  cargo:rustc-env=ZED_COMMIT_SHA=25e2e9c6727ba9b77415588cfa11fd969612adb7
  cargo:rustc-link-arg=/stack:8388608
  cargo:rerun-if-changed=resources/windows/app-icon.ico
  package.metadata.winresource does not exist
  Selected RC path: 'bin\x64\rc.exe'

  --- stderr
  The system cannot find the path specified. (os error 3)
warning: build failed, waiting for other jobs to finish...

为了解决此问题，您可以手动将 ZED_RC_TOOLKIT_PATH 环境变量设置为 RC 工具包路径。通常，您可以将其设置为： C:\Program Files (x86)\Windows Kits\10\bin\<SDK_version>\x64。

有关更多信息，请参阅此 问题。

构建失败：路径过长

构建时您可能会收到类似下面的错误

error: failed to get `pet` as a dependency of package `languages v0.1.0 (D:\a\zed-windows-builds\zed-windows-builds\crates\languages)`

Caused by:
  failed to load source for dependency `pet`

Caused by:
  Unable to update https://github.com/microsoft/python-environment-tools.git?rev=ffcbf3f28c46633abd5448a52b1f396c322e0d6c#ffcbf3f2

Caused by:
  path too long: 'C:/Users/runneradmin/.cargo/git/checkouts/python-environment-tools-903993894b37a7d2/ffcbf3f/crates/pet-conda/tests/unix/conda_env_without_manager_but_found_in_history/some_other_location/conda_install/conda-meta/python-fastjsonschema-2.16.2-py310hca03da5_0.json'; class=Filesystem (30)

为了解决此问题，您可以为 git 和 Windows 启用长路径支持。

对于 git：git config --system core.longpaths true

对于 Windows，使用此 PS 命令：

New-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\FileSystem" -Name "LongPathsEnabled" -Value 1 -PropertyType DWORD -Force

有关更多信息，请参见 win32 文档

（请注意，在启用长路径支持后，您需要重启系统）

图形问题

Zed 启动失败

目前，Zed 在 Windows 上使用 Vulkan 作为其图形 API。然而，Vulkan 在 Windows 上并不总是最可靠的，所以如果 Zed 启动失败，很可能是与 Vulkan 相关的问题。

您可以在以下位置检查 Zed 日志： C:\Users\YOU\AppData\Local\Zed\logs\Zed.log

如果您看到类似下面的消息：

  • Zed failed to open a window: NoSupportedDeviceFound
  • ERROR_INITIALIZATION_FAILED
  • GPU Crashed
  • ERROR_SURFACE_LOST_KHR

那么 Vulkan 可能在您的系统上无法正常工作。在大多数情况下，更新您的 GPU 驱动程序可能有助于解决此问题。

如果日志中没有与 Vulkan 相关的内容，而您恰好安装了 Bandicam，请尝试卸载它。Zed 目前与 Bandicam 不兼容。
