Zed 发布

在这里了解 Zed 的发布渠道。

周三发布流程

您需要对 Zed 仓库具有写入权限才能执行此操作。

此流程中使用的各种服务的凭据可以在 1Password 中找到。

  1. 检出 main 分支并确保您的工作副本是干净的。

  2. 运行 git fetch && git pull 以确保您在本地拥有最新的提交。

  3. 运行 git fetch --tags --force 强制确保您的本地标签与远程同步。

  4. 运行 ./script/get-stable-channel-release-notes。

    • 按照脚本末尾的说明将发布说明聚合到一个结构中。

  5. 运行 ./script/bump-zed-minor-versions。

    • 按照指示推送标签和分支。

  6. 运行 ./script/get-preview-channel-changes。

    • 获取脚本的输出，通过将每个发布说明行组织到相应类别中来构建发布说明。
    • 使用之前的发布作为初始大纲。
    • 如果存在 Credit 行，请确保将其附加到发布说明行的末尾。

  7. 一旦发布草稿在 GitHub Releases 上可用，将预览版和稳定版的发布说明都粘贴到各自的草稿中并保存。

    • 还不要发布草稿。

  8. 检查发布资产。

    • 确保稳定版和预览版的发布任务已经正常完成。
    • 确保每个草稿都有正确数量的资产——发布包目前每个都有10个资产。
    • 下载每个发布草稿的工件并测试您可以在本地运行它们。

  9. 逐一发布稳定版/预览版草稿。

    • 使用 Vercel 检查网站重建的进度。 一旦重建完成，发布将变为公开。

  10. 发布已发送到 Kit 的发布邮件。

    • 确保在发布前仔细检查邮件内容是否正确。
    • 我们有时会修正一些从 GitHub 的渲染器到 Kit 没有正确转换的内容。

  11. 根据稳定版中的热门项目构建社交媒体帖子。

    • 您可以使用上周的帖子链作为您的大纲。
    • 使用 Buffer 为 X 和 BlueSky 两个平台准备文案和资产。
    • 逐一发布，确保两个都发布到各自对应的平台。

补丁发布流程

如果您的 PR 修复了崩溃或崩溃问题，您应该将其 cherry-pick 到当前的稳定版和预览版分支。 如果您的 PR 修复了最近发布代码中的回归问题，您应该将其 cherry-pick 到预览版。

您需要对 Zed 仓库具有写入权限才能执行此操作：

  1. 正常向 main 分支提交包含您更改的 PR。

  2. 一旦被合并，将提交在本地 cherry-pick 到任一发布分支（v0.XXX.x）。

    • 在某些情况下，您可能需要处理合并冲突。 这通常在 cherry-pick 到稳定版分支时发生，因为稳定版分支比预览版分支更加“过时”。

  3. 在 cherry-pick 提交之后，运行 ./script/trigger-release {preview|stable}。 这将提升版本号，创建新的发布标签，并启动发布构建。

    • 这也可以从 GitHub Actions UI 运行：

  4. 一旦发布草稿在 GitHub Releases 上可用，根据需要校对和编辑发布说明并保存。

    • 还不要发布草稿。

  5. 检查发布资产。

    • 确保稳定版/预览版的发布任务已经正常完成。
    • 确保每个草稿都有正确数量的资产——发布包目前每个都有10个资产。
    • 下载每个发布草稿的工件并测试您可以在本地运行它们。

  6. 逐一发布稳定版/预览版草稿。

    • 使用 Vercel 检查网站重建的进度。 一旦重建完成，发布将变为公开。

每夜构建发布流程

除了公开发布之外，我们还有一个鼓励员工使用的每夜构建。 每夜构建通过 cron 每天发布一次，可以按需要频繁发布。 没有发布说明或公告，所以您只需将更改合并到 main 并运行 ./script/trigger-release nightly 即可。
