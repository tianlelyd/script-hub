Rust

Zed 内置了 Rust 支持。

  • Tree-sitter: tree-sitter/tree-sitter-rust
  • 语言服务器: rust-lang/rust-analyzer
  • 调试适配器: CodeLLDB (主要), GDB (次要，不支持 Apple Silicon)

Inlay Hints

以下配置可用于更改 Rust 中 rust-analyzer 的内嵌提示设置：

{
  "lsp": {
    "rust-analyzer": {
      "initialization_options": {
        "inlayHints": {
          "maxLength": null,
          "lifetimeElisionHints": {
            "enable": "skip_trivial",
            "useParameterNames": true
          },
          "closureReturnTypeHints": {
            "enable": "always"
          }
        }
      }
    }
  }
}

更多信息请参见 Rust Analyzer 手册中的内嵌提示部分。

Target directory

可以在 initialization_options 中设置 rust-analyzer 的目标目录：

{
  "lsp": {
    "rust-analyzer": {
      "initialization_options": {
        "rust": {
          "analyzerTargetDir": true
        }
      }
    }
  }
}

设置为 true 将把目标目录设置为 target/rust-analyzer。您也可以使用字符串（如 "target/analyzer"）来代替 true 设置自定义目录。

Binary

您可以配置 Zed 应该使用哪个 rust-analyzer 二进制文件。

默认情况下，Zed 会尝试在您的 $PATH 中查找 rust-analyzer 并使用它。如果该二进制文件能够成功执行 rust-analyzer --help，就会使用它。否则，Zed 会回退到安装自己的稳定版 rust-analyzer 并使用它。

如果您想安装预发布版本的 rust-analyzer，可以在 settings.json 中将 pre_release 设置为 true 来指示 Zed：

{
  "lsp": {
    "rust-analyzer": {
      "fetch": {
        "pre_release": true
      }
    }
  }
}

如果您想禁用 Zed 查找 rust-analyzer 二进制文件，可以在 settings.json 中将 ignore_system_version 设置为 true：

{
  "lsp": {
    "rust-analyzer": {
      "binary": {
        "ignore_system_version": true
      }
    }
  }
}

如果您想使用自定义位置的二进制文件，可以指定 path 和可选的 arguments：

{
  "lsp": {
    "rust-analyzer": {
      "binary": {
        "path": "/Users/example/bin/rust-analyzer",
        "arguments": []
      }
    }
  }
}

这个 "path" 必须是绝对路径。

替代目标

如果您希望 rust-analyzer 为除当前平台以外的目标提供诊断（例如在 macOS 上运行时为 Windows 提供诊断），您可以使用以下 Zed LSP 设置：

{
  "lsp": {
    "rust-analyzer": {
      "initialization_options": {
        "cargo": {
          "target": "x86_64-pc-windows-msvc"
        }
      }
    }
  }
}

如果您使用的是 rustup，可以通过运行以下命令找到可用的目标三元组列表（aarch64-apple-darwin、x86_64-unknown-linux-gnu 等）：

rustup target list --installed

LSP 任务

Zed 使用 tree-sitter 提供任务，但 rust-analyzer 具有通过 LSP 查询文件相关任务的 LSP 扩展方法。 这默认启用，可以如下配置

"lsp": {
  "rust-analyzer": {
    "enable_lsp_tasks": true,
  }
}

手动获取 Cargo 诊断

默认情况下，rust-analyzer 启用了 checkOnSave: true，这会导致每次缓冲区保存都触发 cargo check --workspace --all-targets 命令。 如果使用 checkOnSave: false 禁用（请参见上面服务器配置 json 的示例），仍然可以手动获取诊断，在 Rust 文件中使用 editor: run/clear/cancel flycheck 命令来刷新 cargo 诊断；当设置启用时，项目诊断编辑器也会使用 editor: run flycheck 命令刷新 cargo 诊断。

更多服务器配置

Rust-analyzer 手册描述了 rust-analyzer 语言服务器的各种功能和配置选项。 Zed 中的 Rust-analyzer 使用默认参数运行。

大型项目和性能

在大型项目中可能导致大量资源使用的主要问题之一是以下功能的组合：

rust-analyzer.checkOnSave (默认: true)
    在保存时运行检查命令进行诊断。

rust-analyzer.check.workspace (默认: true)
    是否应该将 --workspace 传递给 cargo check。如果为 false，则会传递 -p <package>。

rust-analyzer.cargo.allTargets (默认: true)
    向 cargo 调用传递 --all-targets

这意味着每次 Zed 保存时，都会运行 cargo check --workspace --all-targets 命令，检查整个项目（工作空间）、lib、doc、test、bin、bench 和其他目标。

虽然这在小项目上运作良好，但在大型项目上扩展性不好。

替代方案是使用任务，因为 Zed 已经提供了 cargo check --workspace --all-targets 任务和在终端输出上 cmd/ctrl 点击导航到错误的能力，并限制或完全关闭保存时检查功能。

保存时检查功能负责基于 cargo check 输出返回部分诊断，因此关闭它将限制 rust-analyzer 使用其自己的诊断。

请考虑从手册中了解更多 rust-analyzer.cargo.、rust-analyzer.check. 和 rust-analyzer.diagnostics. 设置，以进行更精细的配置。 以下是 Zed settings.json 的代码片段（编辑并保存 lsp.rust-analyzer 部分后，语言服务器将自动重启）：

{
  "lsp": {
    "rust-analyzer": {
      "initialization_options": {
        // 从 rust-analyzer 获取更多非 cargo 诊断，
        // 可能包含误报（可以按名称关闭）
        "diagnostics": {
          "experimental": {
            "enable": true
          }
        },
        // 完全禁用检查
        // （忽略下面所有的 cargo 和 check 设置）
        "checkOnSave": false,
        // 仅检查 `lib` 目标
        "cargo": {
          "allTargets": false
        },
        // 在 cargo check 中使用 `-p` 而非 `--workspace`
        "check": {
          "workspace": false
        }
      }
    }
  }
}

多项目工作空间

如果您希望 rust-analyzer 分析同一文件夹中未在 Cargo 工作空间的 [members] 中列出的多个 Rust 项目， 您可以在本地项目设置的 linkedProjects 中列出它们：

{
  "lsp": {
    "rust-analyzer": {
      "initialization_options": {
        "linkedProjects": ["./path/to/a/Cargo.toml", "./path/to/b/Cargo.toml"]
      }
    }
  }
}

代码片段

有一种从 rust-analyzer 获取自定义完成项的方法，它将根据代码片段主体转换代码：

{
  "lsp": {
    "rust-analyzer": {
      "initialization_options": {
        "completion": {
          "snippets": {
            "custom": {
              "Arc::new": {
                "postfix": "arc",
                "body": ["Arc::new(${receiver})"],
                "requires": "std::sync::Arc",
                "scope": "expr"
              },
              "Some": {
                "postfix": "some",
                "body": ["Some(${receiver})"],
                "scope": "expr"
              },
              "Ok": {
                "postfix": "ok",
                "body": ["Ok(${receiver})"],
                "scope": "expr"
              },
              "Rc::new": {
                "postfix": "rc",
                "body": ["Rc::new(${receiver})"],
                "requires": "std::rc::Rc",
                "scope": "expr"
              },
              "Box::pin": {
                "postfix": "boxpin",
                "body": ["Box::pin(${receiver})"],
                "requires": "std::boxed::Box",
                "scope": "expr"
              },
              "vec!": {
                "postfix": "vec",
                "body": ["vec![${receiver}]"],
                "description": "vec![]",
                "scope": "expr"
              }
            }
          }
        }
      }
    }
  }
}

调试

Zed 开箱即用地支持调试 Rust 二进制文件和测试。运行 debugger: start (f4|f4) 来启动这些预配置的调试任务之一。

为了获得更多控制，您可以在 .zed/debug.json 中添加调试配置。请参阅以下示例。

构建二进制文件然后调试

[
  {
    "label": "构建和调试本机二进制文件",
    "build": {
      "command": "cargo",
      "args": ["build"]
    },
    "program": "$ZED_WORKTREE_ROOT/target/debug/binary",
    // 使用 Rust 时，CodeLLDB（非 GDB）需要 sourceLanguages
    "sourceLanguages": ["rust"],
    "request": "launch",
    "adapter": "CodeLLDB"
  }
]

基于构建命令自动定位调试目标

当您使用 cargo build 或 cargo test 作为构建命令时，Zed 可以推断输出二进制文件的路径。

[
  {
    "label": "构建和调试本机二进制文件",
    "adapter": "CodeLLDB",
    "build": {
      "command": "cargo",
      "args": ["build"]
    },
    // 使用 Rust 时，CodeLLDB（非 GDB）需要 sourceLanguages
    "sourceLanguages": ["rust"]
  }
]
