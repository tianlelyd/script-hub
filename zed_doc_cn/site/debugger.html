调试器

Zed 使用 调试适配器协议 (DAP) 为多种编程语言提供调试功能。 DAP 是一个标准化协议，定义了调试器、编辑器和 IDE 之间如何进行通信。 它使 Zed 能够支持各种调试器，而无需实现特定语言的调试逻辑。 Zed 实现协议的客户端部分，各种调试适配器实现服务端部分。

这个协议支持诸如设置断点、单步执行代码、检查变量等功能， 并以一致的方式跨不同编程语言和运行时环境工作。

支持的语言

要调试用特定语言编写的代码，Zed 需要找到该语言的调试适配器。一些调试适配器由 Zed 提供，无需额外设置，另一些则由语言扩展提供。目前以下语言有可用的调试适配器：

  • C (内置)
  • C++ (内置)
  • Go (内置)
  • JavaScript (内置)
  • PHP (内置)
  • Python (内置)
  • Ruby (由扩展提供)
  • Rust (内置)
  • Swift (由扩展提供)
  • TypeScript (内置)

如果你的语言没有列出，你可以通过为其添加调试适配器来贡献代码。查看我们的调试器扩展文档了解更多信息。

点击这些链接获取特定语言和适配器的信息和示例，或继续阅读了解更多适用于所有适配器的 Zed 通用调试功能。

快速开始

对于大多数语言，最快的开始方式是运行 debugger: start (f4|f4)。这会打开新进程模态框，显示当前项目的预配置调试任务的上下文列表。调试任务从测试、入口点（如 main 函数）和其他来源创建 —— 请查阅你的语言文档以获取支持内容的完整信息。

你也可以通过点击调试面板右上角的"加号"按钮来打开同样的模态框。

对于不提供预配置调试任务的语言（包括 C、C++ 和一些扩展支持的语言），你可以在项目根目录的 .zed/debug.json 文件中定义调试配置。这个文件应该是一个配置对象数组：

[
  {
    "adapter": "CodeLLDB",
    "label": "First configuration"
    // ...
  },
  {
    "adapter": "Debugpy",
    "label": "Second configuration"
    // ...
  }
]

查看你的语言文档中涵盖典型用例的示例配置。一旦你向 .zed/debug.json 添加了配置，它们就会出现在新进程模态框的列表中。

如果在 .zed/debug.json 中没有找到配置，Zed 还会从 .vscode/launch.json 加载调试配置，并在新进程模态框中显示它们。

启动和附加

Zed 调试器提供两种调试程序的方式；你可以启动程序的新实例或附加到现有进程。 选择哪种方式取决于你想要实现的目标。

启动新实例时，Zed（以及底层调试适配器）通常能够比附加到现有进程更好地获取调试信息，因为它控制着整个程序的生命周期。 运行单元测试或应用程序的调试版本是启动的良好用例。

与启动相比，附加到现有进程可能看起来较为劣势，但实际情况远非如此；在某些情况下你无法重启程序，例如，错误只能在生产环境中重现，或其他特殊情况。

配置

虽然配置字段依赖于调试适配器，但大多数适配器支持以下字段：

[
  {
    // 调试配置的标签，用于在调试面板和新进程模态框中识别调试会话
    "label": "Example Start debugger config",
    // Zed 应该用来调试程序的调试适配器
    "adapter": "Example adapter name",
    // 请求类型：
    //  - launch: 如果指定了程序，Zed 将启动程序，或显示具有正确配置的调试终端
    //  - attach: Zed 将附加到正在运行的程序进行调试，或当未指定 process_id 时，将显示进程选择器（目前仅支持 node）
    "request": "launch",
    // 要调试的程序。此字段支持使用 ~ 或 . 符号的路径解析。
    "program": "path_to_program",
    // cwd: 默认为项目的当前工作目录 ($ZED_WORKTREE_ROOT)
    "cwd": "$ZED_WORKTREE_ROOT"
  }
]

所有配置字段都支持任务变量。

构建任务

Zed 还允许在 build 字段中嵌入一个在调试器启动前运行的 Zed 任务。这对于设置环境或在调试器启动前运行任何必要的设置步骤很有用。

[
  {
    "label": "Build Binary",
    "adapter": "CodeLLDB",
    "program": "path_to_program",
    "request": "launch",
    "build": {
      "command": "make",
      "args": ["build", "-j8"]
    }
  }
]

构建任务也可以通过未替换的标签引用现有任务：

[
  {
    "label": "Build Binary",
    "adapter": "CodeLLDB",
    "program": "path_to_program",
    "request": "launch",
    "build": "my build task" // Or "my build task for $ZED_FILE"
  }
]

自动场景创建

给定一个 Zed 任务，Zed 可以为你自动创建一个场景。自动场景创建也支持我们从装订线创建场景的功能。 目前 Rust、Go、Python、JavaScript 和 TypeScript 支持自动场景创建。

断点

要设置断点，只需在编辑器装订线中的行号旁边点击。 可以根据需要调整断点；要访问给定断点的其他选项，请右键单击装订线中的断点图标并选择所需的选项。 目前，你可以：

  • 向断点添加日志，当断点被命中时将输出日志消息。
  • 使断点条件化，只有在满足条件时才在断点处停止。条件的语法是特定于适配器的。
  • 向断点添加命中计数，只有在断点被命中一定次数后才会停止。
  • 禁用断点，这将防止其被命中，同时保持在装订线中可见。

一些调试适配器（例如 CodeLLDB 和 JavaScript）还会验证你的断点是否可以被命中；无法被命中的断点在 UI 中会更加显著地显示。

为给定项目启用的所有断点也在调试会话 UI 中的“断点”项目中列出。从 UI 中的“断点”项目你也可以管理异常断点。 调试适配器将在特定类型的异常发生时停止。支持哪些异常类型取决于调试适配器。

设置

调试器的设置在 settings.json 中在 debugger 键下分组：

  • dock: 决定调试面板在 UI 中的位置。
  • stepping_granularity: 决定单步执行的粒度。
  • save_breakpoints: 断点是否应在 Zed 会话之间重用。
  • button: 是否在状态栏中显示调试按钮。
  • timeout: 连接到 TCP 调试适配器时超时错误的时间（毫秒）。
  • log_dap_communications: 是否记录活动调试适配器和 Zed 之间的消息。
  • format_dap_log_messages: 将 DAP 消息添加到调试适配器日志记录器时是否格式化。

停靠

  • 描述：调试面板在 UI 中的位置。
  • 默认值： bottom
  • 设置： debugger.dock

选项

  1. left - 调试面板将停靠在 UI 的左侧。
  2. right - 调试面板将停靠在 UI 的右侧。
  3. bottom - 调试面板将停靠在 UI 的底部。
"debugger": {
  "dock": "bottom"
},

单步执行粒度

  • 描述：调试器将使用的单步执行粒度
  • 默认值： line
  • 设置： debugger.stepping_granularity

选项

  1. Statement - 单步应允许程序运行直到当前语句执行完成。 语句的含义由适配器决定，它可能被视为等同于一行。 例如 'for(int i = 0; i < 10; i++)' 可以被认为包含 3 个语句 'int i = 0'、'i < 10' 和 'i++'。
{
  "debugger": {
    "stepping_granularity": "statement"
  }
}
  1. Line - 单步应允许程序运行直到当前源代码行执行完成。
{
  "debugger": {
    "stepping_granularity": "line"
  }
}
  1. Instruction - 单步应允许一条指令执行（例如一条 x86 指令）。
{
  "debugger": {
    "stepping_granularity": "instruction"
  }
}

保存断点

  • 描述：断点是否应在 Zed 会话之间保存。
  • 默认值： true
  • 设置： debugger.save_breakpoints

选项

boolean values

{
  "debugger": {
    "save_breakpoints": true
  }
}

按钮

  • 描述：是否在调试器工具栏中显示按钮。
  • 默认值： true
  • 设置： debugger.show_button

选项

boolean values

{
  "debugger": {
    "show_button": true
  }
}

超时

  • 描述：连接到 TCP 调试适配器时超时错误的时间（毫秒）。
  • 默认值： 2000
  • 设置： debugger.timeout

选项

integer values

{
  "debugger": {
    "timeout": 3000
  }
}

内联值

  • 描述：是否在调试会话期间启用显示代码中变量值的编辑器内联提示。
  • 默认值： true
  • 设置： inlay_hints.show_value_hints

选项

{
  "inlay_hints": {
    "show_value_hints": false
  }
}

内联值提示也可以从编辑器工具栏的“编辑器控制”菜单中切换。

记录 DAP 通信

  • 描述：是否记录活动调试适配器和 Zed 之间的消息。（用于 DAP 开发）
  • 默认值： false
  • 设置： debugger.log_dap_communications

选项

boolean values

{
  "debugger": {
    "log_dap_communications": true
  }
}

格式化 DAP 日志消息

  • 描述：将 DAP 消息添加到调试适配器日志记录器时是否格式化。（用于 DAP 开发）
  • 默认值： false
  • 设置： debugger.format_dap_log_messages

选项

boolean values

{
  "debugger": {
    "format_dap_log_messages": true
  }
}

自定义调试适配器

  • 描述：自定义程序路径和参数，以覆盖 Zed 启动特定调试适配器的方式。
  • 默认值：适配器特定
  • 设置： dap.$ADAPTER.binary 和 dap.$ADAPTER.args

你可以传递 binary、args 或两者。binary 应该是指向调试适配器（如 lldb-dap）的路径，而不是调试器（如 lldb 本身）。args 设置会覆盖 Zed 否则传递给适配器的任何参数。

{
  "dap": {
    "CodeLLDB": {
      "binary": "/Users/name/bin/lldb-dap",
      "args": ["--wait-for-debugger"]
    }
  }
}

主题

调试器支持以下主题选项：

  • debugger.accent: 用于突出断点和断点相关符号的颜色
  • editor.debugger_active_line.background: 活动调试行的背景颜色

故障排除

如果你在使用调试器时遇到问题，请提交 GitHub 问题或与我们安排上手通话，这样我们可以帮助理解和修复你的问题。

你还可以使用一些功能来收集更多关于问题的信息：

  • 当你在调试面板中运行会话时，可以运行 dev: copy debug adapter arguments 操作来将描述 Zed 如何初始化会话的 JSON 数据复制到剪贴板。这在会话启动失败时特别有用，并且是在提交 GitHub 问题时添加的很好的上下文。
  • 你还可以使用 dev: open debug adapter logs 操作来查看在最近的调试会话中 Zed 与调试适配器的所有通信跟踪。
